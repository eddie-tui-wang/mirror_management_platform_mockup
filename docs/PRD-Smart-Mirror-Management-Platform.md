## 1. 产品概述

### 1.1 产品定义

**Smart Mirror Management Platform** 是一个面向智能试衣镜场景的 **B2B SaaS 后台管理平台**。系统服务于三类用户：

- **平台运营方**：统管全局渠道、客户及资产
- **渠道商**（代理商/经销商）：管理名下客户的开通与账号
- **终端品牌方**（零售客户）：管理门店镜面设备、服装目录与展示模板

系统核心价值：

1. 用统一平台管控分散于全国的智能试衣镜硬件网络
2. 通过三层组织架构和多租户隔离，支持大规模 B2B 分销体系
3. 通过模板内容中央管控 + 客户自主激活，兼顾平台管控力与客户灵活性

### 1.2 业务目标

| 目标 | 衡量指标 |
|------|---------|
| 渠道开通效率 | 渠道管理员开通新客户平均耗时 < 5 分钟 |
| 设备上线效率 | 新设备从上线到被 Acknowledge 耗时 < 1 个工作日 |
| 权限合规 | 越权操作发生率 = 0（三层守卫覆盖全部敏感操作） |
| 数据隔离 | 跨租户数据泄露事件 = 0 |

## 2. 范围定义

### 2.1 功能范围

**In Scope（本次纳入）**

- 三层组织架构：Platform → Channel → Customer
- 三角色 RBAC：PlatformSuperAdmin / ChannelOwner / HQOwner
- 菜单级 / 路由级 / 按钮级三层权限守卫
- 渠道生命周期管理
- 客户管理（Direct / Reseller，Regular / Trial）
- 全局用户管理（平台视图）与组织成员管理（客户视图）
- 服装目录管理（含分类体系与设备分配）
- 模板管理（平台 CRUD + 客户激活）
- **设备管理**（客户端完整流程 + 平台聚合视图）
- **会话管理**（三端只读列表）
- 个人设置（密码修改）

**Out of Scope（本次不纳入）**

- 智能试衣镜端（设备本体）固件 / 操作系统
- AI 试衣渲染引擎与 Prompt 工程细节
- 支付 / 账单 / 发票流程
- 数据分析 / BI 看板（销售统计、设备利用率）
- 邮件 / 短信通知服务的后端实现
- 自定义角色创建（V2）
- 多语言 / 国际化（V2）

## 3. 术语表

| 术语 | 定义 |
|------|------|
| **Platform** | 系统运营方（平台方），拥有全局管理权限 |
| **Channel** | 渠道商 / 代理商 / 经销商；代管下属客户 |
| **Customer** | 终端品牌方（零售商）；管理门店设备与服装 |
| **Direct Customer** | 平台直签客户，不挂靠任何渠道 |
| **Reseller Customer** | 渠道代管客户，归属于某个 Channel |
| **Regular Account** | 正式付费账户，无使用限制 |
| **Trial Account** | 试用账户；受「试用天数」与「销售次数配额」双重限制，任一到达即视为到期 |
| **Master Template** | 平台方创建和管控的模板内容单元 |
| **Template Assignment** | 平台将 Master Template 授权给某客户组织使用的记录 |
| **Device** | 智能试衣镜硬件设备；首次接入系统需由客户管理员 Acknowledge |
| **Acknowledge** | 客户管理员对新上线设备进行确认的操作，设备由"待确认"变为"已管理" |
| **Session** | 用户在某台设备上的一次登录使用记录 |
| **OrgMembership** | 用户与组织之间的角色归属关系 |
| **Portal** | 不同角色对应的管理视图入口（Platform Portal / Channel Portal / Customer Portal） |
| **PermGuard** | 按钮级权限组件；无权限时按钮置灰 + 锁图标 + Tooltip |
| **RouteGuard** | 路由级权限组件；无权限访问时跳转 403 页面 |

## 4. 组织模型

### 4.1 三层层级结构

```
Platform（系统运营方）
├── Channel A — Active
│   ├── Customer Y — Reseller, Regular
│   └── Customer Z — Reseller, Regular
├── Channel B — Disabled
├── Channel C — Active
│   └── Customer V — Reseller, Trial
├── Customer X — Direct, Regular
└── Customer W — Direct, Trial
```

### 4.2 客户类型矩阵

| 维度 | 枚举值 | 说明 |
|------|--------|------|
| **Customer Type** | `Direct` | 平台直签，不属于任何渠道 |
| | `Reseller` | 渠道代管，必须归属一个 Channel |
| **Account Kind** | `Regular` | 正式付费账户 |
| | `Trial` | 试用账户，含天数 + 销售配额限制 |

### 4.3 Trial 到期规则

```
Trial 到期 = (当前日期 ≥ trial_start_date + trial_days)
          OR (trial_used_sales ≥ trial_max_sales)
```

到期后账户状态标记为 `Trial - Expired`，展示到期原因（天数到期 / 配额耗尽）。[默认假设：到期后账户变为只读，现有数据保留，不自动删除设备或服装。]

## 7. 全局 UX 规范

### 7.1 表单校验

- 必填字段：提交时若为空，字段边框变红 + 下方显示错误文案
- 邮箱格式：不符合 RFC 5322 时提示"请输入有效的邮箱地址"
- 唯一性冲突：提交后端返回 409 时，字段内联显示"已存在"提示
- 数字范围：正整数字段输入非正数时阻止提交

### 7.2 操作反馈

| 操作类型 | 反馈方式 |
|---------|---------|
| 成功操作（创建/编辑/启用/禁用） | 顶部 Toast 提示，持续 3 秒 |
| 破坏性操作（删除） | Modal 二次确认 → 成功后 Toast |
| 加载中 | 按钮禁用 + Loading spinner |
| 网络异常 | Toast 错误提示，保留当前状态 |

### 7.3 空状态

所有列表/表格在数据为空时，居中展示空状态图标 + 说明文案。

### 7.4 图片加载

图片加载失败时，展示固定占位图（灰色背景 + 图片图标），不报错。

### 7.5 分页

[默认假设] 所有列表默认每页 10 条，支持切换为 20 / 50 条。

### F-1 全局布局与导航

#### 业务背景

应用外壳（App Shell）为三端 Portal 提供统一的布局结构，根据当前登录角色动态渲染侧边栏菜单、Header 信息与通知入口。

#### 布局结构

```
┌─────────────────────────────────────────────┐
│  Header                                      │
│  [Portal Logo + 名称]        [Bell] [Avatar] │
├──────────┬──────────────────────────────────┤
│ Sidebar  │  Main Content Area               │
│ (Menu)   │                                  │
│          │                                  │
└──────────┴──────────────────────────────────┘
```

#### Header 组件规范

| 元素 | 说明 | 可见角色 |
|------|------|---------|
| Portal 名称 | 当前 Portal 的名称（如"Smart Mirror Platform"） | 全部 |
| 用户头像 + 名称 | 显示当前登录用户姓名与所属组织 | 全部 |
| Avatar 下拉菜单 | 包含"个人设置"和"退出登录"两项 | 全部 |
| 新设备通知 Bell | 显示未 Acknowledge 设备数量的红点计数 | HQOwner |

**Bell 通知行为**：
1. 圆形 Bell 图标，右上角显示红色数字角标（count = 未确认设备数）
2. 数量为 0 时不显示角标
3. 点击 Bell → 跳转至 `/dashboard/customer/devices`
4. 设备页面提供"Mark all as read"操作，点击后清零角标

#### Sidebar 菜单规范

菜单项根据当前角色的权限表动态过滤，无权限项**不渲染**（隐藏，不置灰）。

**平台端菜单**（PlatformSuperAdmin）：
- 渠道管理
- 客户管理
- 用户与角色
- 资产（子菜单展开）
  - 服装库
  - 模板库
  - 设备管理
- 会话管理

**渠道端菜单**（ChannelOwner）：
- 客户管理
- 设备与会话（V2）

**客户端菜单**（HQOwner）：
- 组织成员
- 服装库
- 模板库
- 设备管理
- 会话管理

#### 验收标准（DoD）

- [ ] 登录后侧边栏仅显示该角色有权限的菜单项
- [ ] 不同 Portal 侧边栏顶部色带颜色正确（紫/蓝/绿）
- [ ] HQOwner 有未确认设备时，Bell 正确显示数量角标
- [ ] Avatar 下拉菜单包含"个人设置"和"退出登录"
- [ ] 退出登录后清除 Session，跳转到 `/`

### F-2 认证与登录

#### 业务背景

登录页是系统入口，支持用户以邮箱 + 密码登录系统。登录成功后，系统根据用户角色自动跳转至对应 Portal 首页。Session 在浏览器中持久化，页面刷新不掉登录态。

#### 主流程：登录

**前置条件**：用户持有有效账号
**步骤**：
1. 用户访问 `/`，看到登录表单（邮箱 + 密码 + 登录按钮）
2. 输入凭据，点击登录
3. 系统校验凭据，成功后写入 Session
4. 系统读取用户角色，跳转至对应 Portal 首页

**错误场景**：
- 邮箱格式不合法 → 字段内联提示
- 凭据不匹配 → Toast 提示"邮箱或密码错误"
- 账号被禁用 → Toast 提示"账号已被停用，请联系管理员"

#### Session 管理

| 行为 | 规范 |
|------|------|
| 持久化方式 | localStorage（Key: `smart-mirror-auth`） |
| 刷新页面 | 自动恢复登录态，无需重新登录 |
| 未登录访问内页 | 系统重定向到 `/` |
| 已登录访问登录页 | 允许访问（不自动跳转，方便切换账号） |
| 登出 | 清除 localStorage，跳转 `/` |

| Trial（有效中） | 标签"Trial"，展示"剩余 N 天 / 剩余 M 次配额" |
| Trial（天数到期） | 标签"Trial - Expired"，说明"天数已到期" |
| Trial（配额耗尽） | 标签"Trial - Expired"，说明"销售配额已耗尽" |

#### 边界与异常

| 场景 | 行为 |
|------|------|

#### 验收标准（DoD）

- [ ] 列表仅显示当前渠道下属 Reseller 客户，不显示其他渠道客户或直签客户
- [ ] 账号状态、管理员邮箱、最后登录时间正确内联展示在客户行
- [ ] 从未登录的客户行显示「重发邀请」；已登录过的客户行不显示
- [ ] 重发邀请后 Toast 提示包含目标邮箱
- [ ] Trial 状态展示逻辑与 P-2 一致
- [ ] 创建成功后新客户行立即出现，账号状态和邀请操作正确
- [ ] PlatformSuperAdmin / HQOwner 访问渠道端路由显示 403
- [ ] `/dashboard/channel/users` 路由不再存在（访问返回 404 或重定向至 C-1）

### K-2 客户端·服装库管理

#### 业务背景

HQOwner 管理本组织的服装目录，通过自定义分类体系组织服装，并将服装分配到具体设备上展示。

#### 实体定义：Garment

| 字段 | 说明 | 约束 |
|------|------|------|
| `name` | 服装名称 | 必填 |
| `image` | 产品图片 URL | 必填；加载失败时展示占位图 |
| `category_id` | 所属分类 | 可为空（未分类） |
| `status` | `Active` / `Disabled` | 必填 |
| `owner_org_id` | 所属客户组织 | 必填 |
| `updated_at` | 最后更新时间 | 系统追踪 |

#### 实体定义：GarmentCategory

| 字段 | 说明 | 约束 |
|------|------|------|
| `name` | 分类名称 | 必填；同一组织内唯一 |
| `owner_org_id` | 所属客户组织 | 必填 |
| `garment_count` | 该分类下的服装数 | 系统计算 |

#### 主流程：查看服装列表

**筛选项**：分类（All / 特定分类 / 未分类）
**展示字段**：产品图、服装名称、分类、状态、更新时间、操作列
**操作列**：设置分类 / 编辑 / 移除

#### 主流程：上传 / 导入服装

1. 点击「上传服装」
2. [默认假设] 弹出 Modal，支持填写服装名称和上传图片文件
3. 确认 → 服装添加至本组织目录

#### 主流程：管理分类

1. 点击「管理分类」，弹出分类管理 Drawer
2. **添加分类**：输入分类名称 → 点击添加；若名称与已有分类重复，提示"分类名已存在"
3. **删除分类**：点击分类行删除图标 → Modal 确认：
   - 有关联服装：提示"{N} 件服装将变为未分类"
   - 无关联服装：提示"该分类无关联服装"
   - 确认后：分类删除，关联服装变为"未分类"（服装本身不删除）

#### 主流程：服装设备分配

> 在 K-4 设备详情中实现；亦可从服装行操作触发（见 K-4）。

1. 服装行操作列点击「分配到设备」（[可选配置]：或在 K-4 设备详情中操作）
2. 弹出设备多选列表（展示本组织设备列表）
3. 选择目标设备 → 确认

#### 边界与异常

| 场景 | 行为 |
|------|------|
| 图片加载失败 | 展示占位图 |
| 添加重复分类名 | 提示"分类名已存在"，阻止添加 |
| 删除有服装的分类 | 确认 Modal 展示受影响数量 |
| 筛选"未分类" | 仅展示无分类服装 |

#### 验收标准（DoD）

- [ ] 分类筛选正确过滤列表（含"未分类"筛选）
- [ ] 添加重复分类名被拦截
- [ ] 删除有服装的分类后，相关服装分类字段变为空/未分类
- [ ] 服装图片缺失时占位图展示正常
- [ ] 数据范围严格限制在本组织

### K-4 客户端·设备管理

#### 业务背景

HQOwner 管理本组织的智能试衣镜设备，包括查看设备状态、确认（Acknowledge）新上线设备、设置设备昵称、以及管理每台设备展示的服装列表。新设备接入系统时，Header Bell 图标显示通知，管理员需及时确认。

#### 实体定义：Device

| 字段 | 说明 | 约束 |
|------|------|------|
| `device_id` | 设备唯一标识 / SN | 系统生成 |
| `nickname` | 设备昵称（如"1号店A镜"） | 可选；由 HQOwner 设置 |
| `status` | `Online` / `Offline` | 设备实时上报 |
| `is_acknowledged` | 是否已确认 | 默认 `false`；Acknowledge 后变 `true` |
| `owner_org_id` | 所属客户组织 | 必填 |
| `last_online_at` | 最后在线时间 | 系统追踪 |

#### 实体定义：DeviceGarmentAssignment

| 字段 | 说明 |
|------|------|
| `device_id` | 设备 ID |
| `garment_catalog_id` | 服装 ID |
| `assigned_at` | 分配时间 |

#### 主流程：查看设备列表

**筛选项**：状态（All / Online / Offline）
**展示字段**：设备 ID / SN、昵称、设备状态（色标）、已确认状态、最后在线时间、已分配服装数、操作列
**操作列**：Acknowledge（待确认设备）/ 设置昵称 / 管理服装分配

#### 新设备通知流程

```
设备首次上线
  └── 设备 is_acknowledged = false
      └── Header Bell 角标 +1
          └── HQOwner 点击 Bell → 跳转设备管理页
              └── 点击「Acknowledge」
                  └── is_acknowledged = true
                      └── Bell 角标 -1
```

#### 主流程：Acknowledge 新设备

1. 设备列表中，待确认设备行显示「Acknowledge」按钮（橙色高亮）
2. 点击「Acknowledge」
3. Modal 确认："确认将设备 {device_id} 纳入管理？"
4. 确认 → `is_acknowledged = true`，设备状态正常展示，Bell 角标减 1

**批量 Acknowledge**（可选配置）：在设备页面 Header 区提供「Mark all as read」，一键 Acknowledge 全部待确认设备。

#### 主流程：设置设备昵称

1. 操作列点击「设置昵称」
2. Inline 编辑或 Modal：输入昵称文本（[默认假设] 最多 50 字符）
3. 确认保存 → 列表展示更新

#### 主流程：管理设备服装分配

1. 操作列点击「管理服装」，弹出服装分配 Drawer
2. **已分配区域**：展示该设备已分配的服装列表（含图片缩略图 + 名称），每条有「移除」操作
3. **添加区域**：从本组织服装目录中选择（多选，下拉可搜索）
4. 确认添加 → `DeviceGarmentAssignment` 写入

#### 边界与异常

| 场景 | 行为 |
|------|------|
| 设备列表为空 | 空状态："暂无设备，新设备上线后将在此展示" |
| 无待确认设备 | Bell 不显示角标；设备列表无橙色高亮行 |
| 设备 Offline | 仍可管理服装分配和昵称，但 Offline 状态色标为灰色 |
| 移除服装分配 | Modal 确认："移除后该设备将不再展示此服装" |

#### 验收标准（DoD）

- [ ] 新设备（`is_acknowledged = false`）在列表中高亮显示，Bell 角标正确
- [ ] Acknowledge 操作后 Bell 角标正确减少，设备高亮消失
- [ ] 「Mark all as read」一键确认全部新设备，Bell 角标归零
- [ ] 设备昵称保存后列表即时更新
- [ ] 服装分配 Drawer 展示已分配列表和待选列表，操作正确
- [ ] 数据范围严格限制在本组织设备

### X-1 会话管理

#### 业务背景

会话（Session）记录用户在智能试衣镜设备上的登录使用情况，是运营审计和客户服务的重要数据源。三端提供只读视图，数据范围按角色隔离。

**版本说明**：V2 正式上线，MVP/V1 阶段可作为内测功能。

#### 实体定义：Session

| 字段 | 说明 |
|------|------|
| `session_id` | 会话 ID |
| `user_email` | 登录用户邮箱 |
| `device_id` | 使用的设备 ID |
| `org_name` | 所属组织名称 |
| `status` | `ACTIVE` / `TERMINATED` / `EXPIRED` |
| `started_at` | 会话开始时间 |
| `ended_at` | 会话结束时间（ACTIVE 状态为空） |
| `duration` | 持续时长（系统计算） |

#### 三端视图差异

| Portal | 数据范围 | 路由 |
|--------|---------|------|
| 平台端（X-1a） | 全系统所有会话 | `/dashboard/sessions` |
| 渠道端（X-1b，V2，属 C-2） | 当前渠道下属客户的会话 | `/dashboard/channel/sessions` |
| 客户端（X-1c） | 当前客户组织的会话 | `/dashboard/customer/sessions` |

#### 主流程：查看会话列表

**筛选项**（平台端）：组织（All / 特定组织）、状态（All / ACTIVE / TERMINATED / EXPIRED）
**筛选项**（客户端）：状态（All / ACTIVE / TERMINATED / EXPIRED）
**展示字段**：用户邮箱、设备 ID / 昵称、所属组织（平台端显示）、状态（色标）、开始时间、结束时间、持续时长

> 本模块为**只读**，无操作列。

#### 验收标准（DoD）

- [ ] 平台端展示全量会话，支持组织筛选
- [ ] 客户端仅展示本组织会话
- [ ] 三种 Session 状态色标正确（绿/灰/橙）
- [ ] ACTIVE 状态的会话不显示结束时间，持续时长显示"进行中"
- [ ] 无操作按钮（纯只读）

### X-2 个人设置

#### 业务背景

所有已登录用户均可访问个人设置页，当前 V1 仅提供密码修改功能。

#### 路由

`/dashboard/settings`（三端均可访问）

#### 主流程：修改密码

1. 用户点击 Avatar → 「个人设置」，进入设置页
2. 填写：当前密码（必填）、新密码（必填）、确认新密码（必填）
3. 规则校验：
   - 当前密码不能为空
   - 新密码不能为空，[默认假设] 长度 ≥ 8 字符
   - 确认密码必须与新密码一致
4. 提交 → 系统校验当前密码是否正确 → 成功后更新密码，Toast 提示"密码修改成功"

#### 边界与异常

| 场景 | 行为 |
|------|------|
| 当前密码输入错误 | 提示"当前密码不正确" |
| 两次新密码不一致 | 表单内联提示"两次密码不一致" |
| 新密码与当前密码相同 | 提示"新密码不能与当前密码相同" |

#### 验收标准（DoD）

- [ ] 三端角色均可访问 `/dashboard/settings`
- [ ] 密码长度不足时表单阻止提交
- [ ] 两次密码不一致时内联提示
- [ ] 修改成功后 Toast 反馈正常
- [ ] 修改成功后不强制登出（保持当前 Session 有效）
